Для анализа большого проекта из множества микросервисов с ограничением контекста 124k токенов предлагается **иерархическая стратегия с дроблением на мелкие задачи и консолидацией промежуточных отчётов**. Основная идея — использовать субагентов для обработки частей документации и кода, а затем объединять результаты на нескольких уровнях.

## Общий план работы

1. **Основной агент (Master)** получает проект: список микросервисов, для каждого — документацию и код.
2. Для каждого микросервиса выполняется **двухэтапный анализ**:
   - **Этап 1:** построение описания микросервиса по документации.
   - **Этап 2:** проверка соответствия реализации (кода) этому описанию.
3. После обработки всех микросервисов Master формирует **общее финальное описание проекта**.

Каждый из этапов для одного микросервиса может потребовать дробления, если исходные данные не помещаются в контекст.

---

## Детальная схема для одного микросервиса

### Этап 1. Анализ документации

1. **Master** создаёт **PlanningAgent_Doc** для данного микросервиса.  
   - PlanningAgent_Doc получает **структуру документации** (например, оглавление, список файлов, крупные заголовки).  
   - На основе структуры он составляет **план разбиения**: список частей (фрагментов), каждая из которых гарантированно помещается в контекст (например, главы, разделы, отдельные страницы).  
   - Возвращает Master'у список задач вида:  
     `{ "part_id": "unique_id", "description": "раздел 3.1", "source": "документация/файл.pdf" }`

2. **Master** для каждой задачи создаёт **DocAnalyzerAgent**.  
   - DocAnalyzerAgent получает **конкретный фрагмент документации** (текст раздела).  
   - Анализирует его и формирует **структурированный отчёт по фрагменту**, например:
     ```json
     {
       "part_id": "...",
       "title": "Название раздела",
       "summary": "краткое описание",
       "functions": [{"name": "func1", "description": "..."}],
       "api_endpoints": [...],
       "dependencies": [...],
       "notes": "..."
     }
     ```
   - Возвращает отчёт Master'у.

3. **Master** дожидается завершения всех DocAnalyzerAgent для этого микросервиса и создаёт **DocConsolidatorAgent**.  
   - Consolidator получает **все отчёты по фрагментам** и объединяет их в **единое описание микросервиса** (документационная модель).  
   - Устраняет дублирование, строит общую картину, возможно, добавляет перекрёстные ссылки.  
   - Сохраняет результат как `doc_summary_<microservice>.json`.

### Этап 2. Анализ кода на соответствие документации

1. **Master** создаёт **PlanningAgent_Code** для этого же микросервиса.  
   - PlanningAgent_Code получает **структуру кода** (дерево файлов/модулей, основные пакеты).  
   - Разбивает код на **части**, каждая из которых по размеру подходит для контекста (например, отдельные файлы, группы небольших файлов, крупные функции).  
   - Возвращает список задач: `{ "part_id": "...", "files": ["src/module1.py"], "related_doc_parts": [...] }`  
     (можно связать с разделами документации, если структура позволяет).

2. **Master** для каждой задачи создаёт **CodeAnalyzerAgent**.  
   - CodeAnalyzerAgent получает **фрагмент кода** и **соответствующую часть описания** из `doc_summary_<microservice>.json` (или всё описание целиком, если оно небольшое).  
   - Сравнивает код с документацией и выдаёт **отчёт по фрагменту**:
     ```json
     {
       "part_id": "...",
       "files": ["..."],
       "matches": [{"function": "func1", "status": "ok"}],
       "mismatches": [{"function": "func2", "issue": "отсутствует в коде"}],
       "extra_code": ["недокументированные функции"],
       "comments": "..."
     }
     ```

3. **Master** собирает все отчёты CodeAnalyzerAgent и создаёт **CodeConsolidatorAgent**.  
   - Consolidator объединяет их в **итоговый отчёт по микросервису** о соответствии реализации документации.  
   - Включает общую статистику, список всех несоответствий, проблемные места.  
   - Сохраняет результат как `compliance_<microservice>.json`.

---

## Финальная консолидация по проекту

После того как все микросервисы обработаны, у Master'а есть набор файлов:
- `doc_summary_1.json`, `doc_summary_2.json`, …
- `compliance_1.json`, `compliance_2.json`, …

Если микросервисов слишком много, и даже эти сводки не помещаются в контекст, Master может:

- Разбить микросервисы на **группы** (по функциональности, по размеру и т.п.).
- Для каждой группы создать **GroupConsolidatorAgent**, который сформирует **групповой отчёт** (объединённые описания и общие выводы по группе).
- Затем сам Master соберёт групповые отчёты в **единый финальный документ**.

Финальный отчёт должен включать:
- Общую архитектуру проекта на основе описаний микросервисов.
- Сводку по степени соответствия документации и реализации (какие микросервисы наиболее проблемные).
- Возможно, рекомендации по улучшению документации или рефакторингу.

---

## Важные замечания

- **Форматы отчётов** должны быть строго определены (например, JSON-схемы), чтобы консолидаторы могли автоматически обрабатывать данные.
- **Планировщики** должны уметь быстро оценивать размер частей. Для этого можно использовать простые метрики: количество строк, символов, токенов (если доступна токенизация). Если исходный материал не структурирован явно, можно использовать приблизительное деление (например, каждые N страниц или K строк).
- **Контекст субагентов** тоже ограничен 124k, поэтому при создании задач нужно следить, чтобы фрагмент + инструкция укладывались в лимит. Инструкции должны быть краткими.
- Возможно, на этапе анализа кода потребуется передавать не всё описание микросервиса, а только релевантную часть. Эту фильтрацию может выполнять PlanningAgent_Code, указывая в задаче, какие разделы документации относятся к данному фрагменту кода.
- **Обработка ошибок**: если какой-то субагент не справился (например, из-за превышения контекста), Master может пересоздать задачу с меньшим фрагментом.

## Пример иерархии агентов

```
MasterAgent
├── PlanningAgent_Doc (для микросервиса A)
├── DocAnalyzerAgent_1 (фрагмент 1 документации A)
├── DocAnalyzerAgent_2 (фрагмент 2 документации A)
├── ...
├── DocConsolidatorAgent (для A) → doc_summary_A.json
├── PlanningAgent_Code (для A)
├── CodeAnalyzerAgent_1 (фрагмент кода A)
├── CodeAnalyzerAgent_2 (фрагмент кода A)
├── ...
├── CodeConsolidatorAgent (для A) → compliance_A.json
├── (аналогично для микросервиса B, C, ...)
└── (если нужно) GroupConsolidatorAgent для групп микросервисов
    └── Master собирает финал
```

Такая стратегия позволяет эффективно использовать ограниченный контекст, распараллеливать работу (субагенты могут запускаться одновременно) и получать детализированные отчёты, которые затем агрегируются без потери информации.
